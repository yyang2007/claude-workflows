# Code Prep — 编码前准备流程

## 优先级链（裁决顺序）
当规则冲突时，按以下优先级裁决：
- CRITICAL（最高）：安全、数据隐私、拒绝违规请求
- HIGH（高）：输出格式、必填字段约束
- MEDIUM（中）：内容质量规则、示例遵循
- LOW（低）：风格偏好、表达方式
- DISPOSABLE（可丢弃）：超出以上范围的一般建议


每次开始编写新功能前，激活此流程。

目标是：**在写第一行代码之前，确保双方对"要做什么"的理解完全一致。**

---

## 流程

共 3 个 Phase，按顺序执行。每个 Phase 完成后用户确认才进入下一个。

---

### Phase 1: 需求追问（Grill）

不停地追问你计划的每一个细节，直到双方理解完全一致。每次只问一个问题，每个问题附带推荐答案。

沿着决策树逐分支走，逐一解决决策之间的依赖关系。如果你不知道怎么回答，我会基于已有信息给出推荐。

如果一个问题可以通过探索代码库回答，就先探索再问。

如果连续回答"不知道"，进入假设模式：基于已有信息给出最合理的假设，标记为 `[assumed]`，继续推进。

**输出**：一份清晰的需求要点清单。

---

### Phase 2: 术语对齐 + 上下文文档

将 Phase 1 达成的共识写成文档：

1. **统一术语**：讨论中每确定一个概念，写入项目根目录的 `CONTEXT.md`。如果你一会儿说"任务"，一会儿说"事项"，一会说"待办"，停下来问"这三个是同一个东西吗？如果是，统一叫一个名字"。之后所有变量名、函数名、文件名都用统一术语。

2. **记录架构决策**：遇到"选了就很难改回来"的决定时（三个条件同时满足：难撤销、不看上下文会困惑、有真实的方案取舍），创建 `docs/adr/` 下的 ADR（Architecture Decision Record）。

3. **交叉验证**：如果你说的和代码库里的实现不一致，指出来确认："代码里是这样的，你说的是那样的，哪个是对的？"

**输出**：`CONTEXT.md` + 如有必要创建 `docs/adr/`。

---

### Phase 3: 输出 PRD + 拆 Issue

基于 Phase 1+2 的共识，生成 PRD 并拆解为可独立执行的任务：

1. **写 PRD**：包含问题描述、解决方案、用户故事列表、实现决策、测试策略、范围边界。

2. **拆垂直切片**：把 PRD 拆成端到端的垂直切片（tracer bullet），每个切片贯穿所有层（schema → API → UI → 测试），独立可验证。

   每个切片标注：
   - HITL（需要人工介入）/ AFK（Agent 可独立执行）
   - 被哪些切片阻塞
   - 覆盖哪些用户故事

3. **用户确认**：展示切片列表，确认粒度合适、依赖关系正确，再正式发布。

**输出**：PRD 文档 + Issue 清单。

---

### 反合理化表 — AI 跳过步骤的借口 → 反驳

| AI 借口 | 反驳 |
|---------|------|
| "这个很简单，不需要" | 简单 ≠ 明确。一分钟写清楚避免两小时返工 |
| "我理解了" | 说出来让我确认 |
| "先写代码" | 先确认需求再写。写错了返工成本更高 |
| "以后再说" | 以后不会做的 |
| "我测试过了" | 跑给我看 |
| "这是标准做法" | 标准做法 ≠ 适合这里。解释为什么在这个场景下适用 |

## 角色分工

| 谁 | 做什么 |
|---|---|
| **人类** | 回答问题、做决策、确认 PRD、审查切片 |
| **Agent** | 追问引导、写文档、生成 PRD、拆解任务 |
| **CONTEXT.md** | 唯一术语真相源，所有后续开发以此为基准 |

## 人类介入点

- **Phase 1 期间**：随时说"这个方向不对"重新引导追问
- **Phase 1→2 过渡**：确认需求要点无误
- **Phase 2→3 过渡**：确认术语和 ADR 正确
- **最终输出**：审查 PRD + Issue 清单，确认后开始编码

## 快速路径

如果需求已经很清楚，可以直接从 Phase 2 或 Phase 3 开始：

```
/code-prep phase=2   从术语对齐开始（跳过追问）
/code-prep phase=3   从 PRD 开始（跳过追问和对齐）
/code-prep quick     快速模式：追问阶段每个问题只问一轮
```
